# 四诊综合分析报告单假数据规范

这份文档用于生成管理端“中医四诊综合分析报告单”的 50 份假数据。

目标有两件事：

1. 让生成出来的数据能正常显示在管理端列表、详情、首页统计里。
2. 等你把表格文件给我后，我可以按这份规范把数据导入到管理端对应的数据表。

## 一、建议的数据组织方式

最推荐你给我一个 `xlsx` 文件，里面放 3 个工作表：

1. `reports`
2. `diseases`
3. `attachments`

原因是当前报告数据实际上分成 3 张表：

1. 主表：`tcm_consult_report`
2. 病种表：`tcm_consult_report_disease`
3. 附件表：`tcm_consult_report_attachment`

如果你只给我一张总表，我也能导，但会有两个限制：

1. 每条报告只能方便地带 1 个病种摘要
2. 附件信息会很难保持结构化

所以建议直接按 3 张表准备。

## 二、字段和必填规则

### 1. reports 工作表

每一行代表 1 份报告。

| 列名 | 是否必填 | 说明 | 推荐值/格式 |
|---|---|---|---|
| `report_key` | 必填 | 你自定义的关联键，用来和 diseases、attachments 关联 | `R001`、`R002` |
| `patient_name` | 必填 | 患者姓名 | 中文姓名 |
| `gender` | 必填 | 性别 | `男` / `女` |
| `age` | 必填 | 年龄 | 整数，如 `34` |
| `visit_type` | 必填 | 就诊类型 | `初诊` / `复诊` / `线上问诊` / `线下复诊` |
| `basic_disease` | 可选 | 基础疾病 | 如 `高血压`、`糖尿病` |
| `allergy_history_type` | 可选 | 过敏史类型 | `无` / `有` / `不清楚` |
| `allergy_history_detail` | 可选 | 过敏史详情 | 仅 `有` 时建议填写 |
| `past_history_type` | 可选 | 既往史类型 | `无` / `有` / `不清楚` |
| `past_history_detail` | 可选 | 既往史详情 | 仅 `有` 时建议填写 |
| `current_medication_type` | 可选 | 当前用药类型 | `无` / `有` / `不清楚` |
| `current_medication_detail` | 可选 | 当前用药详情 | 仅 `有` 时建议填写 |
| `chief_complaint` | 建议必填 | 主诉 | 如 `反复胃脘胀痛 2 周，加重 3 天` |
| `duration` | 可选 | 症状持续时间 | 如 `2周`、`3个月` |
| `severity_level` | 可选 | 严重程度 | `1` 到 `5` 的整数 |
| `sudden_worse` | 可选 | 是否突然加重 | `是` / `否` |
| `life_impact` | 可选 | 对生活影响 | 多个值用英文逗号分隔 |
| `highest_risk_level` | 建议必填 | 报告最高风险等级 | `normal` / `medium` / `high` / `critical` |
| `status` | 建议必填 | 报告处理状态 | 见下方状态枚举 |
| `doctor_advice` | 可选 | 医生处理建议 | 非空时可触发“已处理”绿标 |
| `doctor_summary` | 可选 | 医生补充说明 | 非空时可触发“已处理”绿标 |
| `suggested_department` | 可选 | 建议科室 | 如 `中医内科`、`针灸推拿科` |
| `need_offline_visit` | 可选 | 是否建议线下就诊 | `true` / `false` |
| `need_supplement` | 可选 | 是否需要补充资料 | `true` / `false` |
| `create_time` | 建议必填 | 提交时间 | `2026-05-09 10:30:00` |
| `update_time` | 可选 | 更新时间 | `2026-05-09 14:00:00` |
| `doctor_name` | 可选 | 处理医生姓名 | 如 `张医生` |

### 2. diseases 工作表

每一行代表某份报告下的 1 个病种项。

| 列名 | 是否必填 | 说明 | 推荐值/格式 |
|---|---|---|---|
| `report_key` | 必填 | 对应 reports 工作表中的 `report_key` | `R001` |
| `sort_order` | 建议必填 | 排序号 | `1`、`2`、`3` |
| `disease_code` | 可选 | 病种编码 | 如 `A01`、`TCM-NEI-001` |
| `disease_name` | 建议必填 | 病种名称 | 如 `慢性胃炎` |
| `disease_path` | 可选 | 分类路径 | 如 `中医内科/脾胃病/胃脘痛` |
| `risk_level` | 建议必填 | 病种风险等级 | `normal` / `medium` / `high` / `critical` |
| `risk_reason` | 可选 | 风险提示原因 | 如 `伴黑便，建议尽快排查消化道出血风险` |

说明：

1. 同一份报告可以有 1 到 3 条病种记录。
2. 如果 `reports.highest_risk_level` 没手工给，我后续导入时也可以按 diseases 里最高的 `risk_level` 推出来。

### 3. attachments 工作表

每一行代表某份报告下的 1 个附件。

| 列名 | 是否必填 | 说明 | 推荐值/格式 |
|---|---|---|---|
| `report_key` | 必填 | 对应 reports 工作表中的 `report_key` | `R001` |
| `file_type` | 建议必填 | 附件类型 | 见下方附件类型枚举 |
| `file_name` | 建议必填 | 附件名 | 如 `tongue_r001.jpg` |
| `file_url` | 建议必填 | 附件访问地址 | 完整 URL 或可访问路径 |
| `file_size` | 可选 | 文件大小 | 以字节为单位，如 `245678` |

如果暂时没有真实图片/音频，可以先不给附件表，我后续导入时允许为空。

## 三、枚举值建议

### 1. status 状态枚举

管理端当前最适合使用这 5 个值：

| 值 | 含义 |
|---|---|
| `pending` | 待处理 |
| `processing` | 处理中 |
| `supplement_required` | 需补充 |
| `completed` | 已完成 |
| `rejected` | 已驳回 |

其中你这次最关键的是：

1. `pending` 会进入“待处理报告”
2. `processing` / `completed` 更适合视为已处理

### 2. highest_risk_level / risk_level 风险枚举

| 值 | 含义 |
|---|---|
| `normal` | 普通 |
| `medium` | 建议关注 |
| `high` | 高风险 |
| `critical` | 极高风险 |

其中：

1. `high` 和 `critical` 会进入“高风险预警”
2. 首页紧急关注也是按高风险未完成状态在看

### 3. file_type 附件类型枚举

管理端详情页当前识别这些值：

| 值 | 含义 |
|---|---|
| `tongue_image` | 舌象照片 |
| `face_image` | 面部照片 |
| `affected_part_image` | 患处照片 |
| `voice` | 语音描述 |
| `inspection_report` | 检查报告 |
| `other` | 其他附件 |

## 四、哪些字段会影响页面表现

### 1. 待处理报告

首页“待处理报告”与报告单页 `scope=pending` 主要看：

1. `status = pending`

### 2. 今日就诊

首页“今日就诊”与报告单页 `scope=today` 主要看：

1. `create_time` 是否为当天日期

### 3. 高风险预警

首页“高风险预警”与报告单页 `scope=highRisk` 主要看：

1. `highest_risk_level = high`
2. 或 `highest_risk_level = critical`

### 4. 绿色“已处理”标记

当前管理端列表页里，我已经接的规则是：

满足下面任一条件，就显示绿色“已处理”：

1. `doctor_advice` 非空
2. `doctor_summary` 非空
3. `status = processing`
4. `status = completed`

所以如果你想让某一条报告在列表页显示“已处理”，最稳的做法是：

1. 给它填 `doctor_advice`
2. 同时把 `status` 设成 `processing` 或 `completed`

## 五、50 份假数据的推荐分布

为了让管理端首页、列表页和筛选看起来更真实，建议这 50 份数据按下面分布来做：

### 1. 状态分布

| 状态 | 建议数量 |
|---|---|
| `pending` | 18 |
| `processing` | 14 |
| `supplement_required` | 6 |
| `completed` | 10 |
| `rejected` | 2 |

### 2. 风险分布

| 风险等级 | 建议数量 |
|---|---|
| `normal` | 18 |
| `medium` | 16 |
| `high` | 12 |
| `critical` | 4 |

### 3. 时间分布

建议把 50 份数据分布在最近 30 天内，其中：

1. 今天至少 8 到 12 份
2. 最近 7 天占 20 到 25 份
3. 其余分散在最近一个月

这样首页趋势图、今日就诊、病种分布会更像真实业务。

### 4. 病种分布

建议按这些方向分散：

1. 中医内科：20 份左右
2. 针灸推拿科：10 份左右
3. 骨伤科：8 份左右
4. 妇科：6 份左右
5. 儿科：6 份左右

## 六、推荐的 life_impact 可选值

`life_impact` 建议从下面挑，多个值用英文逗号拼接：

1. `影响睡眠`
2. `影响饮食`
3. `影响工作学习`
4. `影响活动`
5. `影响情绪`
6. `无明显影响`

示例：

```text
影响睡眠,影响工作学习
```
