## 一、核心技术类

### Q1：请详细描述项目的整体技术架构。

**回答要点**：
本项目采用**前后端分离 + 微服务架构**，分为三层：

- **用户端（移动端）**：基于 uni-app（Vue3 框架）构建，支持 H5 页面、微信小程序和 Android App 多端发布。负责患者信息采集、舌苔/面部拍照、脉搏记录、声纹录制、问诊报告提交等功能。
- **管理端（医生端）**：基于 Vue3 + Vite + Element Plus + ECharts 构建，桌面端 Web 应用。负责医生查看患者报告、AI 辅助诊断、知识库管理、数据可视化大屏、用户管理等功能。
- **后端（服务端）**：基于 Spring Boot 2.5.14 + MyBatis + Redis + MySQL 的若依（RuoYi）框架，内置 Spring Security 权限体系。提供 RESTful API 和 AI 推理服务。数据库采用 MySQL，Redis 用于 Token 缓存和会话管理。

三个端之间通过 HTTP RESTful API 通信，管理端通过 Vite Proxy 将 `/api` 请求代理到后端 `127.0.0.1:8080`，移动端直接请求后端地址或通过 H5 代理。

### Q2：项目使用了哪些主要的技术栈？为什么选择这些技术？

**回答要点**：

| 技术 | 用途 | 选型理由 |
|---|---|---|
| uni-app | 移动端跨平台框架 | 一套代码同时发布 H5 / 微信小程序 / Android App，降低多端维护成本 |
| Vue3 + Vite | 管理端前端框架 | Vite 开发体验极佳（HMR 冷启动快），Vue3 Composition API 适合复杂逻辑 |
| Element Plus | 管理端 UI 组件库 | 成熟的桌面端组件库，适合医疗管理系统的表格/表单/对话框场景 |
| ECharts | 数据可视化 | 社区活跃、图表类型丰富、支持大数据量渲染和动态更新 |
| Spring Boot | 后端框架 | 成熟的企业级 Java 框架，生态完善，适合复杂业务系统 |
| MyBatis | ORM 框架 | SQL 可控性强，适合中医药领域复杂的关联查询和统计 |
| Redis | 缓存 + Token 存储 | 高性能内存数据库，用于 JWT Token 缓存和会话状态管理 |
| MySQL | 关系型数据库 | 成熟稳定，满足中医诊疗数据的结构化存储需求 |

### Q3：项目中前后端是如何实现数据交互的？

**回答要点**：

前后端通过 **RESTful API + JSON** 格式进行数据交互。具体实现：

1. **管理端**使用 Axios 封装请求实例（`apps/admin-web/src/utils/request.js`），自动添加 JWT Token 到 Authorization 请求头；通过 Vite Proxy 将 `/api` 前缀的请求代理到后端 8080 端口，同时将 `/api/system` 路径映射为 `/system`，`/api` 其他路径映射为根路径。
2. **移动端**使用 uni.request 封装（`apps/mobile-app/common/request.js`），支持 GET/POST 方法和文件上传；Token 存储在 localStorage 中，每次请求自动附带。
3. **交互流程示例**（问诊报告提交）：移动端 4 步骤表单 → `consultReport.js` 构建 Payload → `POST /app/consult/report` → 后端插入数据库 → 返回 reportId → 管理端 `GET /system/consult/report/list` 拉取列表。

### Q4：如何解决跨平台兼容性问题？特别是 uni-app 同时发布 H5 和小程序时的差异处理？

**回答要点**：

项目通过以下方式处理跨平台兼容：

1. **条件编译**（`main.js`）：通过 `#ifndef VUE3` / `#ifdef VUE3` 宏区分 Vue2 和 Vue3 环境，分别使用 `new Vue()` 和 `createSSRApp()` 创建应用实例。
2. **平台差异封装**（`request.js`）：文件上传针对不同平台做了差异化处理——对于 `blob:` 和 `data:` 协议的路径，使用 `fetch + FormData` 方式上传（H5 环境）；对于常规路径，使用 uni-app 的 `uni.uploadFile` 接口（小程序/App 环境）。
3. **API 地址自适应**（`config.js`）：H5 环境下使用相对路径 `/api/system`，通过代理转发；非 H5 环境下使用绝对路径 `http://127.0.0.1:8080/system`。
4. **UI 组件选择**：全部使用 uni-app 内置组件（view/text/image/button），避免使用平台特有组件。

### Q5：项目中使用了哪些设计模式？请举例说明。

**回答要点**：

1. **代理模式**：管理端 Vite 开发服务器的 Proxy 配置，将所有 `/api` 请求代理到后端，解决了前后端分离的跨域问题。
2. **工厂模式**：`consultReport.js` 中的 `buildSubmitPayload()` 方法，将不同来源的表单数据统一构建为后端要求的提交格式。
3. **适配器模式**：`consultReport.js` 中的 `normalizeReportDetail()` 和 `normalizeReportListItem()` 方法，将后端的字段命名（snake_case）转换为前端的字段命名（camelCase），适配不同数据源。
4. **单例模式**：Spring 默认 Bean 作用域（Singleton），`TcmAiChatService`、`TcmRagService` 等 Service 组件全局唯一。
5. **模板方法模式**：若依框架的 `BaseController` 提供统一的 `startPage()` / `getDataTable()` 分页处理逻辑，子 Controller 只需调用即可。
6. **策略模式**：AI 聊天服务中，根据不同的请求参数（`TcmChatRequest`）组装不同的 Prompt 策略，如管理端 AI 使用严格的 System Prompt 约束，App 端使用不同的对话上下文管理方式。

## 二、人工智能与模型类

### Q6：项目中的 AI 功能是如何实现的？用到了哪些 AI 技术？

**回答要点**：

项目中的 AI 功能通过 **大语言模型（DeepSeek API）** 结合 **本地 RAG（检索增强生成）** 实现，具体包括以下技术栈：

1. **大语言模型调用**：通过 `TcmAiChatService` 调用 DeepSeek Chat API（兼容 OpenAI 格式），使用 `deepseek-v4-pro` 模型，支持对话历史管理和系统级 Prompt 约束。
2. **检索增强生成（RAG）**：由 `TcmRagService` 实现，自动扫描项目目录下的知识库文件（JSON/TXT/MD/CSV/SQL），建立 BM25 全文检索索引，在每次 AI 对话时先检索相关知识片段，再作为上下文注入 Prompt。
3. **中文分词与检索**：实现了基于单字 + Bigram（二元组）的中文分词器，支持 CJK 字符和英文单词混合检索，使用 BM25 算法计算相关度评分。
4. **知识库管理**：提供 `GET /system/ai/tcm-kb/status` 查看索引状态和 `POST /system/ai/tcm-kb/reload` 重新加载索引的接口。
5. **体质识别模拟**：移动端舌诊页面集成了面部/舌苔拍照 → AI 体质识别的流程，目前使用 Mock 数据模拟（9种中医体质）。

### Q7：请详细介绍 RAG（检索增强生成）的实现细节。

**回答要点**：

RAG 系统由 `TcmRagService` 实现，核心流程如下：

**第一步：知识发现与索引构建**
- 自动扫描项目目录向上 8 层的常见知识库目录（`data`、`knowledge`、`kb`、`rag`、`database/sql` 等）
- 支持 5 种文件格式：JSON、TXT、MD、CSV、SQL，最大文件限制 25MB
- JSON 文件自动扁平化（`flattenJson` / `flattenJsonNode`），提取键值对
- 中文文本按 **单字符 + Bigram（相邻二元组）** 进行分词，英文单词按字母序列分词
- 每个文档按 800 字符切块（chunk），150 字符重叠，建立 BM25 倒排索引

**第二步：检索与排序**
- 输入查询文本 → 分词 → 计算每个 chunk 的 BM25 分数
- BM25 公式：`score = IDF * (tf * (k1+1)) / (tf + k1 * (1 - b + b * docLen/avgDocLen))`
- 参数 `k1=1.5`, `b=0.75`（标准 BM25 默认值）
- 对结果进行加权调整：含 SQL/DDL 噪声内容降权（×0.12），含临床信号（症状/主治/辨证等）提权（×1.35）
- 去重后按分数降序排列，默认返回 Top 5（可配置为 1-10）

**第三步：上下文注入**
- 将检索结果格式化拼接为"【参考要点N】主题：xxx\n内容：xxx"格式
- 注入到 AI Prompt 的"【知识库检索结果】"段落中
- 约束 AI 回答不能编造知识库中不存在的依据

### Q8：为什么选择 DeepSeek 而不是其他大模型？

**回答要点**：

选择 DeepSeek 基于以下考量：

1. **成本优势**：DeepSeek API 的定价远低于 OpenAI GPT-4 系列，在医疗场景中需要频繁调用 AI 进行辅助分析，长期使用成本可控。
2. **中文能力突出**：DeepSeek 在中文理解、中医领域术语处理上表现优秀，尤其擅长处理文言文和中医经典文献内容。
3. **合规可控**：DeepSeek 为国内模型，数据不出境，符合医疗健康数据的合规要求。
4. **兼容性好**：DeepSeek API 兼容 OpenAI 的 message 格式，便于未来迁移或切换其他模型。
5. **可配置性**：项目在 `application.yml` 中灵活配置 API Key、Base URL 和模型名称，可以通过环境变量 `DEEPSEEK_MODEL` 切换到其他模型。

### Q9：AI Prompt 是如何设计的？如何保证回答的安全性和专业性？

**回答要点**：

AI 系统 Prompt 设计包含以下安全约束机制（详见 `TcmAiChatService.java` 的 `SYSTEM_PROMPT` 常量）：

**角色设定**："中医临床辅助决策助手"，服务对象是**具有执业资质的医生**，不直接面向患者。

**核心约束规则（5条红线）**：
1. **辅助定位**：只能提供辅助参考，不能替代医生诊断
2. **禁止绝对化结论**：不得输出"确诊""必须服用""保证有效"等词语
3. **禁止编造依据**：知识库不足时必须明确写明"当前知识库未检索到充分依据"
4. **强制风险提示**：涉及急症、重症、孕产妇、儿童、老人、肝肾功能异常、抗凝药等场景必须提示复核风险
5. **剂量禁忌**：不得给出超出知识库的精确剂量，不得推荐偏方/秘方/违法治疗

**输出格式规范**：控制在 6 个小节内，格式为：【病例摘要】→【辨证思路】→【可能证型】→【治法与方药参考】→【安全风险】→【建议追问与提醒】

**用户端约束**：在 `buildUserPrompt()` 中额外要求回答"简洁实用""不要展示知识库文件名/片段编号等实现细节"。

### Q10：如果 DeepSeek API 调用失败或超时，系统如何处理？

**回答要点**：

系统设计了三层异常处理：

1. **网络异常层**（`RestClientException`）：捕获连接超时、DNS 解析失败等网络问题，日志记录错误详情（自动脱敏 API Key），抛出 "DeepSeek 服务调用失败，请检查后端网络或稍后重试" 的用户友好提示。
2. **HTTP 状态码异常层**（`RestClientResponseException`）：捕获 401/403/429/500 等服务端错误，日志记录状态码和响应体（脱敏），抛出指引用户检查 API Key 和模型配置的提示。
3. **返回格式解析层**（JSON 解析异常）：如果 API 返回的 JSON 结构不符合预期（如缺少 choices 字段），抛出 "DeepSeek 返回解析失败，请检查接口返回格式" 的提示。

所有异常在 Controller 层通过 `ServiceException` 捕获，转换为统一格式的 `AjaxResult.error()` 返回给前端展示。

### Q11：项目中 AI 的回答为什么会附带知识库引用？这对中医诊断有什么实际意义？

**回答要点**：

AI 回答附带知识库引用（通过 `TcmChatResponse` 的 `references` 字段返回），这在中医临床辅助决策中具有重要价值：

1. **可溯源验证**：中医强调"理法方药"的完整性，AI 给出辨证和方剂建议时需要基于经典文献或临床指南，医生可以追溯到原始知识库内容进行验证。
2. **降低"幻觉"风险**：大模型天生存在幻觉问题，通过 RAG 技术限制模型只能基于知识库内容回答，同时将引用片段展示给医生，让医生可以交叉验证 AI 的建议。
3. **教学辅助价值**：对于实习医生或基层医生，AI 不仅给出结论，还展示依据来源，起到"以案促学"的效果。
4. **质量控制**：知识库管理者可以通过查看引用频率和覆盖度，持续优化知识库的内容质量和检索命中率。

### Q12：BM25 检索算法的具体实现是怎样的？相比其他检索方法有什么优劣？

**回答要点**：

项目实现的 BM25 算法（`TcmRagService.bm25()`）：
- **词频（TF）**：计算查询词在文档块中的出现频率
- **逆文档频率（IDF）**：`log(1 + (总文档数 - 含该词文档数 + 0.5) / (含该词文档数 + 0.5))`
- **长度归一化**：通过 `k1` 和 `b` 参数控制文档长度对评分的影响，长文档不被过度惩罚
- **调优**：使用 `k1=1.5`, `b=0.75`（业界推荐默认值）

**优劣分析**：

| 对比维度 | BM25（本项目） | 向量检索（Embedding） | 混合检索 |
|---|---|---|---|
| 部署复杂度 | 低，纯 Java 实现，无需额外服务 | 高，需要 Embedding 模型和向量数据库 | 最高 |
| 响应速度 | 快（毫秒级） | 中等（需 Embedding 计算） | 中等 |
| 语义理解 | 弱，基于词法匹配 | 强，理解语义相似度 | 最强 |
| 冷启动 | 无需训练，扫描文件即用 | 需要模型部署 | 需额外组件 |
| 中医术语匹配 | 精确命中效果好（如"黄芪"精确匹配） | 近义匹配好（如"气虚"≈"气弱"） | 最好 |

目前的 BM25 实现满足基本需求，未来可升级为 Elasticsearch + 向量检索的混合方案。

### Q13：系统中的体质识别（舌诊/面诊）为什么使用 Mock 数据而不是真实 AI 模型？

**回答要点**：

体质识别模块目前使用 Mock 数据是基于以下阶段性考虑：

1. **先验证流程闭环**：在项目初期，优先打通"患者采集 → 数据上传 → 医生查看"的业务闭环，体质识别结果作为流程中的占位数据，不影响核心业务流的测试和验证。
2. **模型训练数据不足**：高质量的中医舌诊/面诊数据集需要专业中医师标注，获取成本高。市面上的公开数据集（如 CHINATONGUE）在体质的多样性和标注质量上仍有局限。
3. **接口预留与可扩展**：代码结构已经预留了 AI 接口（参见 `photo.vue` 中的 `mockFaceAiData`），Mock 数据完全模拟真实 API 的返回格式（`physique_name`、`physique_analysis`、`risk_warning`、`advices`），未来只需替换 API 端点即可接入真实模型。
4. **竞赛展示策略**：在答辩演示中，Mock 数据可以保证演示效果稳定可控，避免网络和模型推理延迟影响演示流畅度。

## 三、大数据处理类

### Q14：项目中涉及了哪些数据采集和处理环节？

**回答要点**：

本项目作为中医"四诊"数字化平台，涉及四大类数据的采集与处理：

**1. 望诊数据（图像类）**
- 舌苔照片（`tongueImage`）：患者通过手机拍照上传
- 面部照片（`faceImage`）：患者通过手机拍照上传
- 患处照片（`affectedPhoto`）：可选上传
- 处理方式：通过 `uploadCommonFile` 上传至服务器 `/common/upload` 接口，返回文件 URL 存储到数据库

**2. 闻诊数据（音频类）**
- 语音描述（`voiceFile`）：患者长按录音，录制不少于 3 秒的说话音频
- 处理方式：音频文件上传存储，管理端可在线预览播放

**3. 问诊数据（结构化表单）**
- 基本信息：性别、年龄、体重、身高
- 过往病史：checkbox 多选（高血压/糖尿病/心脏病等）
- 主诉信息：病情描述、持续时间、严重程度、突发加重因素、对生活的影响
- 病种选择：通过疾病树选择器（`DiseaseTreeSelector`）选择症状/病种
- 处理方式：结构化 JSON 数据通过 API 提交到后端，持久化到 MySQL

**4. 切诊数据（数值类）**
- 脉搏数据：`pulseRate`（脉搏频率）、`deviceId`（设备标识）、`signalQuality`（信号质量）
- 处理方式：通过 `POST /app/pulse/upload` 上传记录

### Q15：数据库是如何设计的？核心表之间的关系是怎样的？

**回答要点**：

项目数据库基于若依框架扩展，核心业务表集中在 `tcm_vsl.sql` 中：

**核心业务表关系**：

```
tcm_app_user_profile (用户采集信息表)
  ↑ userId
  │
tcm_consult_report (问诊报告主表) ←→ tcm_consult_report_disease (报告-病种关联表)
  ↑ reportId                    ←→ tcm_consult_report_attachment (报告-附件关联表)
  ↑ userId
  │
tcm_pulse_record (脉搏记录表)
  ↑ userId
```

**表结构说明**：

| 表名 | 核心字段 | 说明 |
|---|---|---|
| `tcm_consult_report` | report_id, user_id, patient_name, gender, age, chief_complaint, highest_risk_level, status | 问诊报告主表，记录患者基本信息和主诉 |
| `tcm_consult_report_disease` | id, report_id, disease_code, disease_name, risk_level, risk_reason | 病种关联表，一个报告对应多个病种 |
| `tcm_consult_report_attachment` | id, report_id, file_type, file_name, file_url, file_size | 附件关联表，一个报告对应多个附件（图片/音频） |
| `tcm_app_user_profile` | user_id, sex, age, face_image, tongue_image, voice_file, remark | 用户采集信息表，记录四诊原始数据 |
| `tcm_pulse_record` | id, user_id, pulse_rate, device_id, signal_quality, sampled_at | 脉搏记录表 |

**基础框架表**（`ry_20220822.sql`）：
- `sys_user` / `sys_role` / `sys_menu` / `sys_dept` — 若依标准的 RBAC 权限体系

### Q16：在数据量大的情况下（假设 10 万+ 患者），系统做了哪些性能优化？

**回答要点**：

虽然项目目前处于初期阶段，但在架构层面已经预留了以下优化手段：

1. **分页查询**：所有列表接口都通过若依框架的 `startPage()` + `PageHelper` 插件实现自动分页，管理端报告列表和用户列表均支持页码和每页条数配置。
2. **索引设计**：MySQL 表的主键（`report_id`、`user_id`）建立聚簇索引，预计在未来对 `create_time`（时间范围查询）和 `status`（状态过滤）字段添加辅助索引。
3. **Redis 缓存**：JWT Token、用户会话信息存储在 Redis 中，减少数据库查询压力。
4. **数据静态化**：仪表盘统计数据（Dashboard）目前大部分使用模拟数据，后续可引入定时任务（Quartz）定期聚合统计结果写入缓存表，避免实时 COUNT 大表。
5. **知识库索引内存化**：RAG 的知识库索引存储在 Java 内存中（`ArrayList<IndexedChunk>`），检索速度在毫秒级，避免了每次查询都扫描磁盘文件。

### Q17：问诊报告中的"最高风险等级"是如何计算和判定的？

**回答要点**：

风险等级判定机制如下：

1. **数据来源**：患者在 App 端的"选择病种/症状"步骤（`DiseaseTreeSelector` 组件）中选择的病种列表，每个病种包含用户自评的 `riskLevel` 和 `riskReason` 字段。
2. **等级体系**：4 级风险——`normal`（普通）、`medium`（建议关注）、`high`（高风险）、`critical`（极高风险）。
3. **聚合逻辑**：`buildSubmitPayload()` 在提交时对所有病种的风险等级取**最高值**作为报告的 `highestRiskLevel`；后端 `TcmConsultReportServiceImpl.buildReport()` 确认入库。
4. **前端过滤**：管理端的"紧急关注"面板和风险筛选功能，根据 `highestRiskLevel` 过滤展示高风险/极高风险患者。
5. **统计聚合**：Dashboard 的 `highRiskCount` 通过 `consultReportMapper.countHighRiskUnfinishedReports()` 统计所有未处理的高风险报告。

### Q18：知识库的冷启动问题如何解决？首次部署没有知识文件怎么办？

**回答要点**：

知识库的冷启动问题通过以下设计解决：

1. **自动扫描机制**：`TcmRagService` 实现了 `InitializingBean` 接口，在 Spring 容器启动完成后自动调用 `afterPropertiesSet()` → `reloadIndex()`，扫描预配置的目录路径。
2. **多目录覆盖**：扫描路径包括 `data/`、`knowledge/`、`kb/`、`rag/`、`database/sql/`、`server/ruoyi-backend/sql/` 等 16+ 个预设目录，覆盖项目所有可能存放知识文件的位置。
3. **知识文件推断**：即使不在预设目录中，文件名含 `tcm`、`zhongyi`（中医拼音）、`zhongyao`（中药拼音）、`shennong`、`fangji` 等关键词的文件也会被自动识别并索引。
4. **兜底提示**：当知识库为空时，`search()` 方法抛出明确的错误提示 "本地中医药知识库为空或索引失败，请检查知识库文件路径"。
5. **在线知识源**：README 中提供了开源中医知识库的下载链接（ShenNong_TCM_Dataset，HuggingFace 镜像），方便首次部署时导入。

## 四、系统架构与工程类

### Q19：为什么选择若依（RuoYi）框架作为后端基础？

**回答要点**：

选择若依框架的决策基于以下考量：

1. **开箱即用的权限体系**：若依内置了完整的 RBAC（基于角色的访问控制）权限管理，包括用户管理、角色管理、菜单管理、部门管理，适合医疗系统严格的权限分级需求。
2. **代码生成器**：若依的 generator 模块可以根据数据库表结构自动生成 Controller / Service / Mapper / 前端页面的 CRUD 代码，大幅提升对标准业务表的开发效率。
3. **最佳实践集成**：框架预集成了 Spring Security + JWT、MyBatis + PageHelper、Druid 连接池、Swagger 文档、Redis 缓存等，减少选型和整合成本。
4. **规范统一**：项目采用统一的 RESTful 接口规范（`AjaxResult` 统一返回格式、`TableDataInfo` 分页封装），保证代码风格一致性。
5. **社区活跃**：若依是国内最活跃的 Spring Boot 开源框架之一，文档完善、问题响应快、扩展资源丰富。

### Q20：Spring Security 权限体系在项目中是如何工作的？

**回答要点**：

项目的权限体系基于若依框架的 Spring Security + JWT 方案：

1. **认证流程**：用户登录 → `SysLoginController` 验证用户名密码 → 生成 JWT Token（含用户信息和权限标识）→ 返回前端存储 → 后续请求通过 Authorization 请求头携带 Token。
2. **权限注解**：管理端接口使用 `@PreAuthorize("@ss.hasPermi('system:consultReport:list')")` 注解进行权限校验，确保只有具有相应权限角色的用户才能访问。
3. **Token 管理**：JWT Token 存储在 Redis 中，支持过期时间配置（默认 30 分钟），Token 刷新和踢下线机制由若依框架自动管理。
4. **路由守卫**：前端 Vue Router 通过 `router.beforeEach()` 全局路由守卫检查 Token 是否存在，未登录用户自动重定向到登录页，已登录用户访问登录页自动跳转到首页。
5. **401 处理**：前端请求拦截器中统一处理 401 响应，清除本地 Token 并跳转登录页。

### Q21：移动端（App）和管理端共用同一个后端，接口是如何区分和隔离的？

**回答要点**：

通过 **URL 路径前缀** + **权限控制** 双重机制实现隔离：

1. **URL 路径区分**：
   - `/app/` 前缀——App 端接口（`AppConsultReportController`、`AppUserProfileController` 等）
   - `/system/` 前缀——管理端接口（`TcmConsultReportController`、`TcmAppUserController` 等）
   - `/system/ai/` 前缀——AI 功能接口（`TcmAiController`）

2. **权限控制差异**：
   - App 端接口**不需要** `@PreAuthorize` 权限校验，因为患者通过手机号/密码登录，角色为普通用户
   - 管理端接口**需要** `@PreAuthorize` 权限校验，只有医生/管理员角色才能访问
   - App 端接口中的 `getUserId()` 通过 JWT Token 解析获取当前用户身份

3. **相同业务的不同视角**：
   - 问诊报告：App 端只能查看自己的报告（`selectAppConsultReportDetail()` 按 userId 过滤），管理端可以查看所有报告并修改状态
   - 用户信息：App 端只能查看和编辑自己的 profile，管理端可以查看所有用户并删除

## 五、数据安全与隐私类

### Q23：医疗健康数据非常敏感，项目在数据安全和隐私保护方面做了哪些工作？

**回答要点**：

1. **传输安全**：前端与后端通信统一使用 JWT Token 认证，Token 存储在 localStorage（H5）/ 本地存储（App），请求时通过 Authorization 请求头传递。
2. **存储隔离**：患者的敏感医疗数据（舌苔照片、面部照片、音频文件）上传后存储在服务器指定目录，数据库中仅存储文件 URL 路径而非 Base64 数据。
3. **权限隔离**：App 端用户只能查看自己的问诊报告（`SELECT ... WHERE user_id = ?`），管理端医生需要通过后台登录 + 权限验证才能查看患者完整信息。
4. **API Key 安全**：DeepSeek API Key 仅在后端配置文件中存储（支持环境变量或私有配置文件），不写入前端代码，避免泄露。
5. **日志脱敏**：`TcmAiChatService` 在日志打印时对 API Key 进行脱敏处理（`maskKeyPrefix()` 只显示前 6 位），响应体中的 API Key 也自动替换为 `***`。
6. **就医提醒**：高风险患者报告中自动展示"该类情况可能需要尽快线下就医"的提醒，在信息安全和生命健康之间提供平衡。

### Q24：用户密码是如何存储和验证的？

**回答要点**：

项目使用若依框架标准的密码安全方案：

1. **加密方式**：密码通过 BCrypt 加密算法进行哈希存储，BCrypt 自动添加随机盐值（salt），每次加密结果不同，即使两个用户的密码相同，数据库中的哈希值也不同。
2. **登录验证**：`SysLoginService` 调用 `BCryptPasswordEncoder.matches(rawPassword, encodedPassword)` 验证用户输入的密码与数据库存储的哈希值是否匹配。
3. **登录保护**：配置了密码错误次数限制（默认 5 次），超过次数后账户锁定 10 分钟，防止暴力破解。
4. **传输安全**：若依框架支持 JSEncrypt 前端 RSA 加密传输（`jsencrypt` 依赖已包含），密码在传输过程中进行加密，避免明文泄露。
5. **注册流程**：App 端注册用户通过 `POST /register` 接口，传入 username、password 和 phonenumber，后端使用 BCrypt 加密后写入 `sys_user` 表。

### Q25：患者上传的舌苔照片、面部照片等隐私图像如何保护？

**回答要点**：

1. **物理存储隔离**：图片存储在服务器指定的上传目录（配置项 `ruoyi.profile`，示例路径 `C:/ruoyi/uploadPath`），与应用程序代码分离。
2. **访问控制**：图片通过 `/api/profile/` 路径访问，受后端 Tomcat 静态资源访问控制，不直接暴露给未认证用户。
3. **文件命名**：上传文件使用服务器生成的文件名（UUID + 扩展名），不包含用户标识信息，防止通过文件名推测用户身份。
4. **最小化采集**：在 App 端的引导页明确告知用户"本页面采集的信息，只做本程序健康数据分析使用，我们会严格保密，请放心填写"，做到告知同意。
5. **删除能力**：管理端提供用户信息删除功能（`handleDelete`），可以删除患者的所有采集信息（包括图片），同时保留登录账号。
6. **业务关联性**：附件通过 `report_id` 与问诊报告关联，不直接暴露患者身份。删除报告时附带的图片文件可在数据库层面级联删除。

## 七、创新性与应用价值类

### Q29：项目的核心创新点是什么？

**回答要点**：

1. **"AI + 中医四诊"完整闭环**：区别于市面上只做知识问答或只在管理端做数据分析的产品，本项目实现了"患者端四诊数据采集 → AI 辅助分析 → 医生端诊断决策 → 知识库持续优化"的全流程数字化闭环。

2. **基于 RAG 的中医临床辅助决策**：在通用大模型（DeepSeek）的基础上，叠加了本地中医药知识库的检索增强生成，解决了大模型在中医领域"知识不足、容易幻觉"的核心痛点。同时通过严格的系统 Prompt 约束，确保 AI 回答的安全性和专业性。

5. **四诊数据多模态采集**：同时支持图像（舌苔/面部）、音频（声纹/语音）、结构化表单（问诊）、数值（脉搏）四种模态的数据采集，实现对中医"望闻问切"的完整数字化覆盖。

### Q32：项目中哪些功能使用了大数据技术？如何处理和分析采集到的数据？

### Q37：如何监控系统的运行状态和性能？

5. **Spring Boot Actuator**（可扩展）：若依框架的依赖管理包含 spring-boot-starter-actuator，后续可启用健康检查端点，配合 Prometheus + Grafana 实现生产级监控。

### Q39：项目开发过程中遇到的最大技术挑战是什么？如何解决的？

**回答要点**：

**挑战 1：大模型在中医领域的幻觉问题**
- **问题**：通用大模型（DeepSeek/GPT）在中医领域缺乏专业知识，经常编造方剂、剂量、禁忌。
- **解决方案**：引入 RAG 技术，将本地知识库作为 AI 回答的"事实基础"，同时通过系统 Prompt 强制约束"不得编造""知识库不足时必须明确说明"。
- **权衡**：RAG 增加了系统复杂度和响应延迟，但显著提升了回答的准确性和可追溯性。

**挑战 2：跨平台移动端兼容性问题**
- **问题**：uni-app 在 H5、iOS/Android App、微信小程序上的 API 行为不一致（特别是文件上传、录音、本地存储）。
- **解决方案**：在 `request.js` 中针对不同平台做了差异化处理——H5 使用 `fetch + FormData`，App 使用 `uni.uploadFile`，通过条件编译做环境隔离。
- **权衡**：部分实现需要写平台特定的分支代码，增加了维护成本。

**挑战 3：中医术语的分词和检索**
- **问题**：常规的 IK Analyzer 等分词器对中医术语（如"肝阳上亢""气滞血瘀""脾肾阳虚"）的分词效果不佳。
- **解决方案**：自研了基于单字符 + Bigram（相邻二元组）的分词器，对 CJK 字符按字切分并组合相邻二元组，英文单词按字母序列提取。这种方法虽然朴素但适合中医专业文本。
- **权衡**：Bigram 分词会产生更多无意义组合，但通过 BM25 的 TF-IDF 机制，高频的组合自然获得更高权重。

**挑战 4：前后端数据模型不一致**
- **问题**：后端数据库字段是 snake_case（如 `patient_name`），前端 JavaScript 习惯使用 camelCase（如 `patientName`）。
- **解决方案**：前端通过 `normalizeReportDetail()` / `normalizeReportListItem()` 等适配器函数进行字段映射转换，后端 `buildSubmitPayload()` 反向转换。
- **权衡**：增加了额外的前端数据处理层，但保持了后端数据库规范的独立性。

## 十二、评委追问与开放性类

### Q42：如果要求你在一周内给项目增加一个"AI 辅助开方"功能，你会如何设计？

**回答要点**：

设计思路：

**第一步：安全设计（最重要）**
- AI 只能推荐方剂参考，不能直接开具处方（法规限制）
- 所有推荐必须标注"仅供参考，需执业中医师审核"
- 剂量范围采用知识库内的标准范围，严禁 AI 自行调整
- 增加配伍禁忌检查（十八反十九畏数据库）

**第二步：功能流程**
1. 医生输入辨证结果（或 AI 辅助辨证后确认的证型）
2. AI 根据证型 + 知识库检索，推荐多个方剂候选（经典方 + 经验方）
3. 每个候选方剂展示：方剂名称、组成药物、剂量参考、功效、禁忌
4. 医生可以"采纳"某个方剂作为基础，再手动加减
5. 最终处方需要电子签名确认后生效

**第三步：数据支持**
- 建立方剂知识库：包含《伤寒论》《金匮要略》等经典方剂 + 现代临床经验方
- 建立配伍禁忌库：十八反十九畏、妊娠禁忌、证候禁忌
- 建立药物剂量库：不同年龄/体质的剂量调整规则

**第四步：技术实现**
- 新增 `TcmPrescriptionController` 提供方剂查询和推荐 API
- 扩展 RAG 系统，增加方剂和配伍数据的索引
- 管理端新增"AI 开方"组件，嵌入现有报告详情页

### Q43：你们的数据量目前看起来不大，如果系统推广到 100 家基层诊所，每天产生 10 万条问诊记录，架构需要做什么调整？

**回答要点**：

从小规模到大规模需要做以下架构升级：

**数据库层**：
1. **分库分表**：按诊所 ID 或时间维度进行水平分片（可使用 ShardingSphere 或 MyCat）
2. **读写分离**：主库写入 + 从库查询，使用 MySQL 主从复制
3. **归档策略**：历史数据定期归档（3 个月以上的已处理报告迁移到归档表或 ClickHouse）

**缓存层**：
1. **多级缓存**：本地 Caffeine 缓存 + Redis 分布式缓存
2. **缓存策略**：热门数据（常用知识库片段/标准病种树）预加载到缓存，减少数据库查询

**检索层**：
1. **从 BM25 升级到 Elasticsearch**：替代内存中的 BM25 索引，支持更大规模的知识库
2. **引入向量检索**：使用 Embedding 模型生成文本向量，支持语义相似度检索

**服务层**：
1. **微服务拆分**：将 AI 对话、知识库检索、报告处理拆分为独立服务
2. **消息队列**：使用 RabbitMQ / RocketMQ 异步处理文件上传、报告状态变更等非实时操作
3. **AI 服务异步化**：AI 对话改为 SSE（Server-Sent Events）流式输出，提升用户体验

**监控运维**：
1. **引入 APM**：SkyWalking / Pinpoint 链路追踪
2. **自动化告警**：Prometheus + AlertManager

### Q44：如果 DeepSeek API 服务关闭或政策变化，你们有什么备选方案？

**回答要点**：

设计了多层次的可替代方案：

1. **模型层切换**（热切换）：通过环境变量 `DEEPSEEK_MODEL` 和 `DEEPSEEK_BASE_URL` 即可切换到其他兼容 OpenAI 接口的大模型，如通义千问 Qwen、智谱 GLM、百度文心、ChatGPT 等。API 调用代码兼容 OpenAI 格式，迁移成本低。

2. **离线知识库兜底**：当大模型不可用时，RAG 系统仍然可用，可以直接展示知识库检索结果作为 AI 替代方案。管理端可以仅展示检索到的相关知识片段，供医生自行参考。

3. **本地小模型方案**（中远期）：考虑在本地部署 Qwen2.5-7B / DeepSeek-R1-Distill-Qwen 蒸馏模型，使用 llama.cpp / ollama 等推理框架，在本地 GPU 或 CPU 上运行。性能会下降但完全不受外部 API 影响。

4. **规则引擎辅助**（最低兜底）：开发基于规则的中医辅助系统——通过决策树 + 知识库构建基础辨证推荐，虽然不如大模型灵活，但稳定可靠、零外部依赖。

### Q45：项目的商业模式是什么？如何实现可持续运营？

**目标客户**：基层中医诊所、社区卫生服务中心、中医院校

**商业模式**（三阶段）：

- 中医继续教育课程集成

**成本控制**：
- API 调用成本：通过 RAG 减少大模型 Token 消耗（只传递检索结果而非完整文档），配合缓存机制降低重复查询
- 开源技术栈：全栈使用开源组件（Spring Boot / Vue / uni-app / MySQL），零许可证费用

### Q46：项目的开源策略是什么？

**回答要点**：

项目基于若依框架（MIT 协议）构建，若依框架本身就是开源项目。我们目前的策略是：

1. **代码开放**：项目代码在 Git 仓库中管理，README 提供了完整的部署文档和目录结构，便于其他人复现和使用。
2. **模块分离**：核心功能与业务逻辑分离——若依框架为基础底座，中医业务模块为可扩展插件。
3. **标准化接口**：所有 API 接口遵循 RESTful 规范，便于第三方集成和二次开发。
4. **知识库开放**：推荐使用的公开数据集（ChatMed_TCM）在 HuggingFace 上免费开放，降低使用门槛。
5. **文档完善**：README 提供了项目结构说明、快速启动指南、环境变量说明、默认端口列表和验证步骤，帮助开发者快速上手。

### Q47：你们认为未来中医数字化的发展方向是什么？项目在这条路上处在什么位置？

**回答要点**：

**中医数字化发展方向**：

```
阶段一（当前）→ 阶段二（近期）→ 阶段三（中期）→ 阶段四（远期）
  信息化         数字化          智能化          智慧化
  ──────┼──────────┼──────────────┼──────────────┼─────▶
         ↑
     本项目定位
```

- **阶段一（信息化）**：电子病历、HIS 系统，以记录为主。已完成。
- **阶段二（数字化）**：四诊客观化采集、结构化存储。**本项目处于此阶段。**
- **阶段三（智能化）**：AI 辅助辨证、方剂推荐、疗效预测。项目通过 RAG + LLM 已经初步探索此阶段。
- **阶段四（智慧化）**：中医"治未病"个性化健康管理、中西医结合智能决策。属于远期展望。

**项目当前位置**：处于 **阶段二为主，向阶段三过渡** 的位置。核心优势在于：
- 已经完成了四诊数据的数字化采集和闭环管理（阶段二的核心目标）
- AI 辅助辨证和知识库检索正在从"辅助参考"走向"临床可用"（阶段三的探索）
- 但距离真正可临床落地的 AI 辨证还有距离（需要更大规模的高质量标注数据）

### Q48：做中医 AI 最大的难点是什么？你们如何应对？

**回答要点**：

**最大难点：中医的"主观性"与 AI 的"客观性"之间的鸿沟**

具体体现：
1. **辨证标准不统一**：同一个患者，不同中医师可能有不同的辨证结论（如：肝郁脾虚 vs 肝胃不和），导致训练数据的标注一致性差。
2. **数据稀缺性**：高质量的"四诊数据 + 金标准诊断结果"数据集极其稀缺（公开数据集极少，且多为非结构化文本）。
3. **隐性知识**：中医诊断中存在大量"只可意会不可言传"的隐性知识（如脉诊的"浮、中、沉"），难以量化为 AI 可学习的特征。

**我们的应对策略**：

1. **技术层面**：采用 RAG + LLM 而非端到端模型——不追求 AI 替代医生辨证，而是让 AI 做"知识检索和参考建议"，医生做最终决策。这样规避了"AI 辨证不准确"的风险。
2. **产品层面**：定位为"辅助工具"而非"诊断工具"，AI 的建议永远标注"仅供参考"，系统中也明确提示"高风险情况需尽快线下就医"。
3. **数据积累策略**：通过 App 端的信息采集逐步积累患者数据，形成"数据飞轮"——先用规则和通用模型提供服务，在服务过程中积累高质量中医诊疗数据，为后续模型训练做准备。
4. **人机协作设计**：AI 提供建议，医生确认/修改，系统记录医生的最终判断和反馈，形成"人在回路中"（Human-in-the-Loop）的持续优化机制。

### Q49：项目中 AI 回答的安全质量如何保障？如果 AI 给出错误建议导致医疗事故，责任如何划分？

**回答要点**：

**技术保障**（防线级）：
1. **第一道防线 — Prompt 约束**：系统 Prompt 明确设定 AI 角色为"辅助决策助手"，禁止绝对化结论（"确诊""必用""保证有效"），必须在回答开头和结尾声明仅供参考。
2. **第二道防线 — RAG 知识约束**：AI 只能基于本地知识库内容回答，降低幻觉概率。知识库不足时必须明说，不得编造。
3. **第三道防线 — 高风险提示**：涉及急症、重症、特殊人群（孕产妇/儿童/老人）时，强制提示"重点复核风险"。
4. **第四道防线 — 剂量安全**：禁止给出超出知识库依据的精确剂量，禁止推荐偏方/秘方。

**产品保障**：
1. **角色隔离**：AI 面向的是**执业医师**而非患者，医生有专业判断能力对 AI 建议进行审核。
2. **流程设计**：AI 建议仅作为参考，处方和诊断必须由医生签字确认（系统设计中预留了 `doctorId`、`doctorName`、`doctorAdvice` 等字段）。
3. **免责声明**：AI 页面明确标注"仅作参考，不可替代医生诊断"。

**法律与伦理**：
- 目前 AI 辅助诊断在国内属于辅助角色，最终诊断责任由执业医师承担。系统设计中始终把 AI 定位为"辅助工具"而非"诊断主体"。
- 对于竞赛答辩而言，重点是阐明团队已经充分认识到这一风险，并在系统设计中做出了多层防护，体现了负责任的技术态度。

### Q50：演示中的 AI 对话看起来响应很快，背后是怎么实现的？能否现场展示修改 Prompt 观察回答变化？

**回答要点**：

**响应速度优化**：
1. **RAG 预检索**：用户输入查询时，先通过 BM25 引擎在内存索引中毫秒级检索相关知识片段，然后拼接 Prompt 调用 DeepSeek API。
2. **API 配置优化**：`max_tokens=1600`（限制输出长度）、`temperature=0.15`（低温度使回答更稳定，减少不必要的随机变化），`readTimeout=60000`（60 秒超时）。
3. **历史消息裁剪**：`MAX_HISTORY_MESSAGES=6` 限制上下文轮数，`MAX_HISTORY_CONTENT_LENGTH=800` 限制每轮消息字数，避免超长对话导致 API 延迟增长。

**现场演示 Prompt 修改**：
可以现场展示修改 `TcmAiChatService.java` 中的 `SYSTEM_PROMPT` 常量或 `buildUserPrompt()` 方法，观察回答的变化。例如：
- 去掉"只能提供辅助参考"约束 → AI 回答可能变得更绝对、更肯定
- 修改输出格式要求（如只输出 3 个小节）→ AI 回答结构随之改变
- 调整 `temperature=0.8` → 回答变得更有创意但可能更不稳定

⚠️ 注意：演示 Prompt 修改时需考虑 API 调用耗时（约 2-5 秒），建议提前准备好对比结果截图。
